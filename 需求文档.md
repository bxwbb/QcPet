# QcPet 需求文档

## 1. 项目背景

QcPet 是一个运行在 Paper 服务端上的全服宠物插件，目标是为服务器提供可配置、可持久化、可扩展的宠物养成能力。插件需要兼顾单服部署的简单性和多服共享数据的可运维性。

## 2. 项目目标

- 为玩家提供稳定的宠物拥有、查看、切换、展示和养成体验。
- 支持数据库存储和本地文件存储，满足不同规模服务器的部署需求。
- 提供高度可配置的宠物定义、行为规则、消息文本和权限控制。
- 保证主要数据操作可靠，避免因服务端重启、玩家下线或数据库异常导致数据丢失。
- 为后续接入经济、任务、活动或跨服系统预留清晰边界，但不提前实现未确认功能。

## 3. 用户角色

| 角色 | 说明 |
| --- | --- |
| 普通玩家 | 使用宠物、查看宠物、参与养成玩法 |
| 管理员 | 发放、移除、修复玩家宠物数据，重载配置 |
| 服主/运维 | 配置存储方式、备份数据、调整玩法参数 |
| 开发者 | 维护插件代码，扩展宠物行为或对接外部系统 |

## 4. 功能需求

### 4.1 宠物基础功能

- 玩家可以拥有一个或多个宠物。
- 玩家可以查看自己的宠物列表。
- 玩家可以选择当前出战或展示的宠物。
- 玩家下线、切服或服务端重启后，宠物数据应保持一致。
- 插件应能限制玩家最大宠物数量。

### 4.2 宠物养成

- 宠物应支持基础属性，例如等级、经验、名称、类型和状态。
- 插件应支持配置经验获取规则。
- 插件应支持配置等级成长规则。
- 插件应支持配置互动或成长冷却，防止刷取。
- 宠物成长结果应可持久化。

### 4.3 宠物展示与行为

- 宠物可跟随玩家或以插件配置的方式展示。
- 玩家进入不可用世界、离线或权限不足时，应隐藏或卸载宠物实体。
- 插件应处理玩家传送、死亡、切换世界等场景。
- 服务端关闭或插件卸载时，应正确清理运行中的宠物实体和缓存。

### 4.4 存储能力

- 插件必须支持本地文件存储。
- 插件必须支持数据库存储。
- 插件应通过统一存储接口隔离具体存储实现。
- 数据保存应尽量异步执行，避免阻塞主线程。
- 插件应支持启动时加载数据、运行时更新数据、关闭时刷盘。
- 数据库不可用时，应输出明确错误日志，并按配置决定是否禁用插件或降级处理。

### 4.5 配置能力

- 插件应支持通过配置文件控制存储方式。
- 插件应支持配置宠物定义、成长规则、消息文本和权限限制。
- 插件应在配置缺失或格式错误时给出明确日志。
- 插件应提供配置重载能力；不适合热重载的配置项应明确提示需要重启。

### 4.6 命令需求

| 命令类型 | 需求说明 |
| --- | --- |
| 玩家命令 | 打开宠物界面、查看宠物、切换宠物、隐藏/显示宠物 |
| 管理命令 | 发放宠物、移除宠物、查询玩家宠物、重载配置 |
| 调试命令 | 可选，仅在开发或管理员权限下查看缓存、存储状态和运行状态 |

具体命令名称和参数以 `plugin.yml` 或命令注册实现为准。

### 4.7 权限需求

- 普通玩家权限应与管理权限分离。
- 不同宠物类型、数量上限和管理能力应可通过权限控制。
- 权限检查失败时应返回可配置消息。

## 5. 非功能需求

### 5.1 性能

- 不应在主线程执行耗时数据库读写。
- 宠物刷新、保存和跟随逻辑应有频率控制。
- 缓存应有明确生命周期，避免玩家离线后长期占用内存。

### 5.2 可靠性

- 玩家数据保存失败时应记录日志。
- 服务端关闭时应尝试保存未写入数据。
- 本地数据文件应避免部分写入导致损坏。
- 数据库表结构升级应有兼容策略。

### 5.3 可维护性

- 宠物领域逻辑、命令、配置、存储和实体控制应分层。
- 数据库实现和本地文件实现应依赖同一抽象接口。
- 配置读取应集中处理，避免在业务代码中散落硬编码键名。
- 消息发送应统一入口，便于后续多语言或格式化扩展。

### 5.4 安全性

- 数据库密码等敏感配置不应输出到普通日志。
- 管理命令必须校验权限。
- 玩家输入的宠物名称、命令参数和配置内容应做长度与格式限制。

## 6. 推荐模块划分

| 模块 | 职责 |
| --- | --- |
| `QcPetPlugin` | 插件生命周期入口 |
| `PetManager` | 玩家宠物状态、缓存和业务协调 |
| `PetService` | 宠物领域规则，例如发放、切换、升级 |
| `PetRepository` | 持久化接口 |
| `DatabasePetRepository` | 数据库存储实现 |
| `FilePetRepository` | 本地文件存储实现 |
| `PetConfig` | 宠物定义和规则配置 |
| `MessageService` | 消息格式化和发送 |
| `PetCommand` | 命令解析与权限校验 |
| `PetEntityController` | 宠物实体生成、跟随、隐藏和清理 |

## 7. 数据模型建议

### 7.1 玩家宠物数据

| 字段 | 说明 |
| --- | --- |
| `player_uuid` | 玩家 UUID |
| `pet_id` | 宠物唯一实例 ID |
| `pet_type` | 宠物类型 |
| `name` | 宠物显示名称 |
| `level` | 等级 |
| `experience` | 经验 |
| `active` | 是否当前启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 7.2 宠物类型配置

| 字段 | 说明 |
| --- | --- |
| `type` | 宠物类型标识 |
| `display_name` | 显示名称 |
| `entity_type` | 实体类型或展示方式 |
| `permission` | 使用权限 |
| `max_level` | 最大等级 |
| `growth_rule` | 成长规则引用 |

## 8. 验收标准

- 插件可在目标 Paper 版本正常加载和卸载。
- 本地存储模式下，玩家宠物数据在重启后不丢失。
- 数据库存储模式下，多次重启后数据一致。
- 普通玩家不能执行管理员命令。
- 配置错误时日志能定位具体配置项。
- 玩家下线、传送、切换世界和服务端关闭时，不残留异常宠物实体。
- 重载配置后，可热更新的配置项生效；不可热更新项有明确提示。

## 9. 约束与边界

- 当前需求不强制实现 GUI，除非插件已有界面设计。
- 当前需求不强制接入经济、任务或跨服消息系统。
- 当前需求不要求实现宠物战斗 AI，除非后续明确提出。
- 当前需求优先保证核心宠物数据、展示和存储稳定。

## 10. 待确认事项

- 目标 Minecraft 与 Paper 版本。
- Java 版本和构建工具。
- 实际支持的数据库类型。
- 具体命令、权限节点和配置文件结构。
- 是否需要 GUI 菜单。
- 是否需要经济插件、权限插件或 PlaceholderAPI 对接。
